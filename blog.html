Event Capture vs Event Bubbling

As per MDN:

In the capturing phase:
The browser checks to see if the element‚Äôs outer-most ancestor (<html>) has an onclick event handler registered on it for the capturing phase, and runs it if so.
Then it moves on to the next element inside <html> and does the same thing, then the next one, and so on until it reaches the element that was actually selected.
In the bubbling phase: the exact opposite occurs:
The browser checks to see if the element selected has an onclick event handler registered on it for the bubbling phase, and runs it if so.
Then it moves on to the next immediate ancestor element and does the same thing, then the next one, and so on until it reaches the <html> element.

Difference between SVG & CANVAS

Posted 31 stJuly 2018

When to use what?

Often, I kept wondering if canvas should be used instead of svg. But I did not give much thought to it until I had to ask this question in an Interview. As per one of the stack overflow answer these are the major difference:

Canvas

  • Pixel based (Dynamic .png)
  • Single HTML element.(Inspect element in Developer tool. You can see only canvas tag)
  • Modified through script only
  • Event model/user interaction is granular (x,y)
  • Performance is better with smaller surface, a larger number of objects (>10k), or both

SVG

  • Shape based
  • Multiple graphical elements, which become part of the DOM
  • Modified through script and CSS
  • Event model/user interaction is abstracted (rect, path)
  • Performance is better with smaller number of objects ( <10k), a larger surface, or both

Advantage of using SVG would be that resolution remains the same across platforms which lacks in Canvas. Google maps, earlie used to rely on svg, now they are canvas.

The SVG grid system considers top left as its default position

Posted 11 thJanuary 2018

Is that a big deal?

Yes? Because since kids we were taught that 0,0 starts at bottom left and that is how we draw graphs all over. But cruel HTML engines draws its page starting from top left as 0,0 and then calculates pixels accordingly.

As per mdn:
For all elements, SVG uses a coordinate system or grid system..That is, the top left corner of the document is considered to be the point (0,0), or point of origin

Did that change your perception a bit? I did mine. I spent couple of hours figuring out why my d3 was plotting graphs from top left. wish I had know this before.

1. HTML5 Is a Work in Progress

As cool as it is to see what HTML5 can do for you, it hasn't been standardized like HTML4. You don't have to worry about updating pages built using HTML4. It's more than ten years old and it's a set standard. If you jump into HTML5 with both feet, you're going to be making updates. Elements and attributes are added and modified several times a year. Of course, this is dependent how much you depend on rich elements, but it's certainly a risk you must take into consideration when using a fluid language. Build with HTML4, play with HTML5.

2. Simplified Syntax
